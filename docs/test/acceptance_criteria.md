# Acceptance Criteria

## 1. Overview

本ドキュメントは PromptNotes の受入基準を定義する。PromptNotes は AI へ渡すプロンプトを素早く書き溜めるローカルノートアプリであり、Tauri (Rust + WebView) + Svelte + CodeMirror 6 で構成され、データは `.md` ファイルとしてローカルファイルシステムに保存される。

対象プラットフォームは Linux および macOS。Windows は将来対応のためスコープ外とする。

本受入基準は `req:promptnotes-requirements` を単一信頼源として導出し、E2E テストの基盤となる。すべての受入基準項目は、後続の E2E テスト生成メタプロンプトにおいてテストシナリオへマッピングされる。

### 非交渉条件への準拠

| 条件 ID | 対象モジュール | 制約内容 | 本ドキュメントでの反映箇所 |
|---|---|---|---|
| Convention 1 | `module:editor` | Cmd+N / Ctrl+N 即時新規ノート作成および1クリックコピーボタンはコアUX。未実装ならリリース不可。 | AC-EDIT-01, AC-EDIT-02, AC-EDIT-06 |
| Convention 2 | `module:editor` | CodeMirror 6 必須。タイトル入力欄禁止・Markdownプレビュー（レンダリング）禁止。実装した場合リリース不可。 | AC-EDIT-03, FC-SCOPE-01, FC-SCOPE-02 |
| Convention 3 | `module:storage` | ファイル名規則 `YYYY-MM-DDTHHMMSS.md` および自動保存は確定済み。違反時リリース不可。 | AC-STOR-01, AC-STOR-02, AC-STOR-03 |
| Convention 4 | `module:feed` | デフォルト直近7日間フィルタ・タグ/日付フィルタ・全文検索は必須機能。未実装ならリリース不可。 | AC-FEED-01, AC-FEED-02, AC-FEED-03, AC-FEED-04 |

## 2. Acceptance Criteria

### module:editor — エディタ機能

#### AC-EDIT-01: キーボードショートカットによる新規ノート作成 [リリースブロッキング]

- **前提**: アプリが起動しトップフィード画面が表示されている
- **操作**: Cmd+N (macOS) / Ctrl+N (Linux) を押下する
- **期待結果**:
  1. フィード最上部に新規ノートカードが編集モードで挿入される
  2. CodeMirror 6 エディタインスタンスが即座に出現する
  3. フォーカスが新規カードの本文入力領域に自動移動する
  4. 対応する `.md` ファイルがデフォルト保存ディレクトリに作成される
- **計測基準**: ショートカット押下からエディタ出現まで 200ms 以内

#### AC-EDIT-02: New ボタンによる新規ノート作成 [リリースブロッキング]

- **前提**: アプリが起動しトップフィード画面が表示されている
- **操作**: ヘッダーの New ボタンをクリックする
- **期待結果**: AC-EDIT-01 と同一の結果が得られる

#### AC-EDIT-03: CodeMirror 6 エディタの使用 [リリースブロッキング]

- **前提**: ノートカードが編集モードに遷移した状態
- **期待結果**:
  1. エディタが CodeMirror 6 インスタンスであること（`.cm-editor` DOM 要素が存在する）
  2. タイトル入力欄が存在しないこと
  3. Markdown レンダリング（プレビュー）パネルが存在しないこと

#### AC-EDIT-04: Markdown シンタックスハイライト

- **前提**: 編集モードのノートカード
- **操作**: `# 見出し` や `**太字**` などの Markdown 記法を入力する
- **期待結果**: Markdown トークンに対応するシンタックスハイライトが適用される（`.cm-header`, `.cm-strong` などの CSS クラスが付与される）
- **否定条件**: Markdown がレンダリング（HTML 変換表示）されないこと

#### AC-EDIT-05: frontmatter 領域の視覚的区別

- **前提**: frontmatter を含むノートを編集モードで開いている
- **期待結果**: `---` で囲まれた frontmatter 領域が本文と異なる背景色で表示される

#### AC-EDIT-06: 1クリック Copy ボタン [リリースブロッキング]

- **前提**: 本文が入力されたノートカードが表示モードで存在する
- **操作**: Copy ボタンを1回クリックする
- **期待結果**:
  1. ノートの本文全体（frontmatter を除く）がシステムクリップボードにコピーされる
  2. コピー成功のフィードバック（ボタン状態変化またはトースト）が表示される

#### AC-EDIT-07: Delete ボタンによるノート削除

- **前提**: ノートカードが存在する
- **操作**: Delete ボタンをクリックする
- **期待結果**:
  1. 対象ノートカードがフィードから消える
  2. 対応する `.md` ファイルがファイルシステムから削除される

#### AC-EDIT-08: 表示モードから編集モードへの遷移

- **前提**: ノートカードが表示モードで表示されている
- **操作**: カードをクリックする
- **期待結果**:
  1. カードが編集モードに切り替わる
  2. CodeMirror 6 エディタが出現し、ノートの現在の内容が表示される
  3. frontmatter 領域が背景色で区別される

#### AC-EDIT-09: 編集モードから表示モードへの遷移と自動保存

- **前提**: ノートカードが編集モードで表示されている
- **操作**: カード外の領域をクリックする（または別のカードをクリックする）
- **期待結果**:
  1. 編集中のカードが表示モードに戻る
  2. 変更内容が対応する `.md` ファイルに自動保存される
  3. 表示モードで更新後の内容が反映されている

#### AC-EDIT-10: 同時編集カードの排他制御

- **前提**: カード A が編集モードで表示されている
- **操作**: カード B をクリックする
- **期待結果**:
  1. カード A が自動保存され表示モードに戻る
  2. カード B が編集モードに切り替わる
  3. 同時に編集モードのカードは 1 つだけである

#### AC-EDIT-11: 新規カードのフィード最上部挿入

- **前提**: 既存ノートが複数表示されている
- **操作**: 新規ノートを作成する
- **期待結果**: 新規カードがフィードの最上部（最初の位置）に表示される

#### AC-EDIT-12: 新規カード作成時のフォーカス自動移動

- **前提**: 新規ノートが作成された直後
- **期待結果**: キーボードフォーカスが新規カードの CodeMirror 6 本文入力領域にある（`document.activeElement` が `.cm-content` 内である）

### module:storage — ストレージ機能

#### AC-STOR-01: ファイル名規則 `YYYY-MM-DDTHHMMSS.md` [リリースブロッキング]

- **前提**: 新規ノートが作成される
- **期待結果**: 生成されるファイル名が正規表現 `^\d{4}-\d{2}-\d{2}T\d{6}\.md$` に合致する
- **例**: `2026-04-13T143205.md`

#### AC-STOR-02: 自動保存によるファイル永続化 [リリースブロッキング]

- **前提**: 編集モードでノートの内容を変更した
- **操作**: カード外をクリックして編集モードを終了する
- **期待結果**:
  1. 対応する `.md` ファイルの内容が変更後の内容と一致する
  2. ファイルの書き込みがアプリのクラッシュなしに完了する

#### AC-STOR-03: frontmatter 形式の正しさ [リリースブロッキング]

- **前提**: タグ `gpt`, `coding` を持つノートが保存される
- **期待結果**: ファイル先頭が以下の YAML frontmatter 形式である
  ```
  ---
  tags: [gpt, coding]
  ---
  ```
- **制約**: frontmatter に含まれるメタデータは `tags` のみ。作成日はファイル名から取得する。

#### AC-STOR-04: OS 別デフォルト保存ディレクトリ

- **前提**: 初回起動時に保存ディレクトリが未設定
- **期待結果**:
  - Linux: `~/.local/share/promptnotes/notes/`
  - macOS: `~/Library/Application Support/promptnotes/notes/`

#### AC-STOR-05: 設定からの保存ディレクトリ変更

- **前提**: 設定モーダルで保存ディレクトリを変更する
- **操作**: 新しいディレクトリパスを入力して保存する
- **期待結果**:
  1. 以降の新規ノートが新しいディレクトリに作成される
  2. 新しいディレクトリ内の既存 `.md` ファイルがフィードに表示される

### module:feed — フィード表示・検索・フィルタ機能

#### AC-FEED-01: デフォルト直近7日間フィルタ [リリースブロッキング]

- **前提**: アプリ起動直後のトップフィード画面
- **期待結果**:
  1. ファイル名のタイムスタンプが過去7日間以内のノートのみ表示される
  2. 7日より前のノートは表示されない
- **境界値**: 丁度7日前の 00:00:00 のノートは表示される

#### AC-FEED-02: 降順表示 [リリースブロッキング]

- **前提**: 複数のノートが存在する
- **期待結果**: フィード上でノートがファイル名タイムスタンプの降順（新しい順）に並ぶ

#### AC-FEED-03: タグ・日付フィルタ [リリースブロッキング]

- **タグフィルタ**:
  - **操作**: 特定のタグ（例: `gpt`）でフィルタする
  - **期待結果**: frontmatter の `tags` に `gpt` を含むノートのみ表示される
- **日付フィルタ**:
  - **操作**: 特定の日付範囲を指定する
  - **期待結果**: ファイル名のタイムスタンプが指定範囲内のノートのみ表示される

#### AC-FEED-04: 全文検索 [リリースブロッキング]

- **前提**: 複数のノートが保存されている
- **操作**: 検索バーにキーワードを入力する
- **期待結果**:
  1. ノート本文にキーワードを含むノートのみ表示される
  2. 検索はファイル全走査で実行される（外部インデックスエンジン不要）

#### AC-FEED-05: スクロールによる過去ノート閲覧

- **前提**: デフォルトフィルタ範囲内のノートが画面に収まらない量存在する
- **操作**: フィードを下方向にスクロールする
- **期待結果**: 画面外のノートカードが順次表示される

### module:ui — UI 全般

#### AC-UI-01: 設定モーダル

- **操作**: ヘッダーの ⚙️ ボタンをクリックする
- **期待結果**:
  1. モーダルダイアログが開く
  2. 保存ディレクトリの変更フォームが表示される

#### AC-UI-02: アプリ名非表示

- **前提**: トップフィード画面が表示されている
- **期待結果**: ヘッダーにアプリ名テキストが表示されていない（画面領域をノートに最大限使う）

#### AC-UI-03: 表示モードのカード内容

- **前提**: ノートカードが表示モードである
- **期待結果**: カードに以下が表示される
  1. 本文のプレーンテキスト
  2. タグ一覧
  3. タイムスタンプ（ファイル名由来）

## 3. Failure Criteria

以下の条件のいずれかに合致した場合、リリースを不可とする。

### FC-SCOPE-01: タイトル入力欄の存在 [リリースブロッキング]

- **条件**: 新規ノート作成画面または編集モードにタイトル入力欄（`<input>` や専用テキストフィールド）が存在する
- **根拠**: Convention 2 — タイトル入力欄禁止はスコープ外として明示されており、実装した場合リリース不可
- **検出方法**: DOM 内にノートタイトル用の入力要素が存在しないことをアサーション

### FC-SCOPE-02: Markdown プレビュー（レンダリング）の存在 [リリースブロッキング]

- **条件**: Markdown を HTML にレンダリングして表示するプレビューパネルまたはモードが存在する
- **根拠**: Convention 2 — Markdownプレビュー（レンダリング）禁止
- **検出方法**: 編集モード中に `.cm-editor` 以外の Markdown レンダリング出力（`<h1>`, `<strong>` など Markdown 由来の HTML タグ）が表示されないことをアサーション

### FC-SCOPE-03: AI 呼び出し機能の存在

- **条件**: アプリ内から外部 AI API を呼び出す機能（ボタン、メニュー、自動トリガー）が存在する
- **検出方法**: ネットワークリクエストの監視で AI API エンドポイントへの通信がないこと

### FC-STOR-01: ファイル名規則違反 [リリースブロッキング]

- **条件**: 生成されたノートファイルの名前が `YYYY-MM-DDTHHMMSS.md` 形式に合致しない
- **検出方法**: 保存ディレクトリ内の全ファイル名を正規表現 `^\d{4}-\d{2}-\d{2}T\d{6}\.md$` で検証

### FC-STOR-02: 自動保存失敗 [リリースブロッキング]

- **条件**: 編集モード終了後にファイルが更新されない、または編集内容とファイル内容が不一致
- **検出方法**: 編集→表示モード遷移後にファイル読み込みで内容を比較

### FC-EDIT-01: Cmd+N / Ctrl+N 未応答 [リリースブロッキング]

- **条件**: キーボードショートカット Cmd+N (macOS) / Ctrl+N (Linux) を押下しても新規ノートが作成されない
- **検出方法**: ショートカット押下後にフィード最上部に編集モードのカードが存在しないことで判定

### FC-EDIT-02: Copy ボタン未動作 [リリースブロッキング]

- **条件**: Copy ボタンクリック後にクリップボードにノート本文がコピーされない
- **検出方法**: Clipboard API の読み取りでコピー内容を検証

### FC-FEED-01: デフォルトフィルタ欠如 [リリースブロッキング]

- **条件**: アプリ起動直後に全期間のノートが表示される（直近7日間フィルタが適用されていない）
- **検出方法**: 7日より前のタイムスタンプを持つテストノートが初期表示に含まれていないことをアサーション

### FC-FEED-02: 検索・フィルタ未実装 [リリースブロッキング]

- **条件**: 全文検索バー、タグフィルタ、日付フィルタのいずれかが機能しない
- **検出方法**: 各フィルタ操作後の表示ノート数とフィルタ条件の整合性を検証

### FC-EDITOR-01: CodeMirror 6 未使用 [リリースブロッキング]

- **条件**: 編集モードで CodeMirror 6 インスタンス（`.cm-editor` DOM 要素）が検出されない
- **根拠**: Convention 2 — CodeMirror 6 必須

## 4. E2E Test Generation Meta-Prompt

### 目的

本セクションは `codd propagate` による E2E テスト自動生成の機械可読命令である。受入基準（セクション2）および失敗基準（セクション3）からテストシナリオを導出し、Playwright を用いて Tauri アプリケーションの E2E テストを生成する。

### テストランタイム環境

#### アプリケーション起動シーケンス

1. `npm run build` または `pnpm build` で Svelte フロントエンドをビルドする
2. `cargo tauri dev` で Tauri 開発サーバーを起動する（テスト時はヘッドレス WebView）
3. ヘルスチェック: WebView が ready 状態になるまで最大 30 秒待機する
4. CI 環境では `xvfb-run` (Linux) を使用して仮想ディスプレイ上で実行する

#### ブラウザ設定

```typescript
{
  headless: true,
  timeout: 30000,
  viewport: { width: 1280, height: 720 }
}
```

Tauri アプリのため、Playwright の Electron 統合モード (`_electron`) または WebDriver 経由で WebView に接続する。`package.json` の `scripts` および `tauri.conf.json` からエントリポイントを自動検出すること。

### MECE ドメイン分割

### テストレベル分離

#### API 統合テスト（Tauri コマンドレベル）

PromptNotes は Web アプリケーションではなく Tauri デスクトップアプリのため、API 統合テストは HTTP エンドポイントではなく Tauri IPC コマンド（`invoke`）を対象とする。

- Tauri の `invoke` コマンド呼び出しによるファイル CRUD 操作のレスポンスを検証する
- ファイル名規則、frontmatter 形式、保存ディレクトリパスなどのデータ契約を検証する
- ファイルシステムの状態（ファイル存在、内容一致）を直接アサーションする

ファイル命名規則: `tests/e2e/<domain>.spec.ts`

#### ブラウザテスト（WebView 操作レベル）

- Tauri WebView 内の実際の DOM を操作する：ボタンクリック、テキスト入力、キーボードショートカット押下
- 各ユーザー操作後、結果として表示される UI 状態（DOM 要素の存在・内容・スタイル）をアサーションする
- ページ遷移を伴う操作（設定モーダル開閉など）では、遷移後の表示要素を必ず検証する。HTTP ステータスのみの確認は不十分とする
- サーバーヘルスベースライン: Tauri バックエンドへの IPC 呼び出しが例外を返さないことを、ビジネスロジックのアサーションに先立って検証する

ファイル命名規則: `tests/e2e/<domain>.browser.spec.ts`

### シナリオ導出ルール

1. **正常系**: 各受入基準 (AC-*) の「期待結果」を1つ以上のテストケースに変換する。複数ステップの状態遷移（例: 表示モード → クリック → 編集モード → 外部クリック → 自動保存 → 表示モード）はチェーン全体の各リンクを個別にアサーションする
2. **異常系**: 各失敗基準 (FC-*) を反転し、禁止状態が発生しないことをアサーションするテストケースとする
3. **境界値**: AC-FEED-01 の7日間境界、AC-STOR-01 のタイムスタンプ秒精度を境界値テストとして追加する
4. **排他制御**: AC-EDIT-10 の同時編集排他は、連続するカード切り替え操作で検証する

### アーキテクチャ適応ルール

- テスト生成時に `src-tauri/src/` 内の Tauri コマンド定義と `src/` 内の Svelte コンポーネント構造をスキャンする
- 未実装のコマンドまたはコンポーネントに依存するテストは `test.fixme()` でマークし、スキップではなく明示的に未実装であることを記録する
- `tauri.conf.json` の `allowlist` からアプリが使用する Tauri API を検出し、テストのセットアップに反映する
