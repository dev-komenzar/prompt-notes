# Acceptance Criteria

## 1. Overview

本ドキュメントは PromptNotes の受入基準を定義する。PromptNotes は AI へ渡すプロンプトを素早く書き溜めるローカルノートアプリであり、Tauri (Rust + WebView) + Svelte + CodeMirror 6 で構成され、データは `.md` ファイルとしてローカルファイルシステムに保存される。

対象プラットフォームは Linux および macOS。Windows は将来対応のためスコープ外とする。

本受入基準は `req:promptnotes-requirements` を単一信頼源として導出し、E2E テストの基盤となる。すべての受入基準項目は、後続の E2E テスト生成メタプロンプトにおいてテストシナリオへマッピングされる。

### 非交渉条件への準拠

| 条件 ID | 対象モジュール | 制約内容 | 本ドキュメントでの反映箇所 |
|---|---|---|---|
| Convention 1 | `module:editor` | Cmd+N / Ctrl+N 即時新規ノート作成および1クリックコピーボタンはコアUX。未実装ならリリース不可。 | AC-EDIT-01, AC-EDIT-02, AC-EDIT-06, AC-EDIT-06b |
| Convention 2 | `module:editor` | CodeMirror 6 必須。タイトル入力欄禁止・Markdownプレビュー（レンダリング）禁止。実装した場合リリース不可。 | AC-EDIT-03, FC-SCOPE-01, FC-SCOPE-02 |
| Convention 3 | `module:storage` | ファイル名規則 `YYYY-MM-DDTHHMMSS.md` および自動保存は確定済み。違反時リリース不可。 | AC-STOR-01, AC-STOR-02, AC-STOR-03, AC-STOR-06 |
| Convention 4 | `module:feed` | デフォルト直近7日間フィルタ・タグ/日付フィルタ・全文検索は必須機能。未実装ならリリース不可。 | AC-FEED-01, AC-FEED-02, AC-FEED-03, AC-FEED-04 |

## 2. Acceptance Criteria

### module:editor — エディタ機能

#### AC-EDIT-01: キーボードショートカットによる新規ノート作成 [リリースブロッキング]

- **前提**: アプリが起動しトップフィード画面が表示されている
- **操作**: Cmd+N (macOS) / Ctrl+N (Linux) を押下する
- **期待結果**:
  1. フィード最上部に新規ノートカードが編集モードで挿入される
  2. CodeMirror 6 エディタインスタンスが即座に出現する
  3. フォーカスが新規カードの本文入力領域に自動移動する
  4. 対応する `.md` ファイルがデフォルト保存ディレクトリに作成される
- **計測基準**: ショートカット押下からエディタ出現まで 200ms 以内

#### AC-EDIT-02: New ボタンによる新規ノート作成 [リリースブロッキング]

- **前提**: アプリが起動しトップフィード画面が表示されている
- **操作**: ヘッダーの New ボタンをクリックする
- **期待結果**: AC-EDIT-01 と同一の結果が得られる

#### AC-EDIT-03: CodeMirror 6 エディタの使用 [リリースブロッキング]

- **前提**: ノートカードが編集モードに遷移した状態
- **期待結果**:
  1. エディタが CodeMirror 6 インスタンスであること（`.cm-editor` DOM 要素が存在する）
  2. タイトル入力欄が存在しないこと
  3. Markdown レンダリング（プレビュー）パネルが存在しないこと

#### AC-EDIT-04: Markdown シンタックスハイライト

- **前提**: 編集モードのノートカード
- **操作**: `# 見出し` や `**太字**` などの Markdown 記法を入力する
- **期待結果**: Markdown トークンに対応するシンタックスハイライトが適用される（`.cm-header`, `.cm-strong` などの CSS クラスが付与される）
- **否定条件**: Markdown がレンダリング（HTML 変換表示）されないこと

#### AC-EDIT-05: frontmatter 領域の視覚的区別

- **前提**: frontmatter を含むノートを編集モードで開いている
- **期待結果**: `---` で囲まれた frontmatter 領域が本文と異なる背景色で表示される

#### AC-EDIT-06: 表示モードでの 1クリック Copy ボタン [リリースブロッキング]

- **前提**: 本文が入力されたノートカードが表示モードで存在する
- **操作**: Copy ボタンを1回クリックする
- **期待結果**:
  1. Copy ボタンが表示モードのカード内に常時描画されている
  2. ノートの本文全体（frontmatter を除く）がシステムクリップボードにコピーされる
  3. コピー成功のフィードバック（ボタン状態変化またはトースト）が表示される
- **否定条件（視認性・クリック可能性）**:
  - Copy ボタンの bounding box の幅・高さがいずれも 0 ではないこと（DOM に存在しても視覚的に潰れていないこと）
  - 親要素の `overflow` 制約や flex/grid の縮小によって視覚的にクリップされていないこと
  - `getBoundingClientRect()` で得られる中心座標で `document.elementFromPoint()` がボタン要素または子要素を返すこと（他要素に被覆されていないこと）
  - フィード内のノート件数が増えても上記条件が維持されること（カードが縦方向に圧縮されないこと）

#### AC-EDIT-06b: 編集モードでの 1クリック Copy ボタン [リリースブロッキング]

- **前提**: 本文が入力されたノートカードが編集モードに遷移している（CodeMirror 6 エディタが出現している）
- **操作**: Copy ボタンを1回クリックする
- **期待結果**:
  1. Copy ボタンが編集モードのカード内に常時描画されている（表示 → 編集の遷移でアンマウントされない）
  2. 現在 CodeMirror 6 に表示されている **未保存を含む最新の本文全体**（frontmatter を除く）がシステムクリップボードにコピーされる
  3. コピー成功のフィードバック（ボタン状態変化またはトースト）が表示される
  4. コピー操作前後で、編集モードが維持される（表示モードに戻らない）
  5. コピー操作は自動保存を誘発しない（ファイル更新時刻は変化しない）
  6. コピー操作は CodeMirror 6 のフォーカス・テキスト選択範囲・Undo 履歴を変更しない
- **否定条件**:
  - Copy ボタンが編集モードで非表示・無効・別レイアウトで省略されていないこと
  - Copy ボタンの bounding box の幅・高さがいずれも 0 ではないこと（DOM に存在しても視覚的に潰れていないこと）
  - カード本体や親フィードコンテナの `overflow` 制約により視覚的にクリップされていないこと
  - `getBoundingClientRect()` で得られる中心座標で `document.elementFromPoint()` がボタン要素または子要素を返すこと（CodeMirror エディタ等に被覆されていないこと）

#### AC-EDIT-07: Delete ボタンによるノート削除

- **前提**: ノートカードが存在する
- **操作**: Delete ボタンをクリックする
- **期待結果**:
  1. 確認ダイアログは表示されず即時削除される
  2. 対象ノートカードがフィードから消える
  3. 対応する `.md` ファイルがファイルシステムから削除される
- **否定条件（視認性・クリック可能性）**:
  - Delete ボタンの bounding box の幅・高さがいずれも 0 ではないこと
  - 親要素の `overflow` やレイアウト圧縮で視覚的にクリップされていないこと
  - `getBoundingClientRect()` の中心座標で `document.elementFromPoint()` がボタン要素または子要素を返すこと
  - フィード内のノート件数が増えても上記条件が維持されること

#### AC-EDIT-08: 表示モードから編集モードへの遷移

- **前提**: ノートカードが表示モードで表示されている
- **操作**: カードをクリックする
- **期待結果**:
  1. カードが編集モードに切り替わる
  2. CodeMirror 6 エディタが出現し、ノートの現在の内容が表示される
  3. frontmatter 領域が背景色で区別される

#### AC-EDIT-09: 編集モードから表示モードへの遷移と自動保存

- **前提**: ノートカードが編集モードで表示されている
- **操作**: カード外の領域をクリックする（または別のカードをクリックする）
- **期待結果**:
  1. 編集中のカードが表示モードに戻る
  2. 変更内容が対応する `.md` ファイルに自動保存される
  3. 表示モードで更新後の内容が反映されている

#### AC-EDIT-10: 同時編集カードの排他制御

- **前提**: カード A が編集モードで表示されている
- **操作**: カード B をクリックする
- **期待結果**:
  1. カード A が自動保存され表示モードに戻る
  2. カード B が編集モードに切り替わる
  3. 同時に編集モードのカードは 1 つだけである

#### AC-EDIT-11: 新規カードのフィード最上部挿入

- **前提**: 既存ノートが複数表示されている
- **操作**: 新規ノートを作成する
- **期待結果**: 新規カードがフィードの最上部（最初の位置）に表示される

#### AC-EDIT-12: 新規カード作成時のフォーカス自動移動

- **前提**: 新規ノートが作成された直後
- **期待結果**: キーボードフォーカスが新規カードの CodeMirror 6 本文入力領域にある（`document.activeElement` が `.cm-content` 内である）

#### AC-EDIT-13: 編集モード遷移時のカーソル初期位置

- **前提**: 以下のいずれかの操作で EditMode に遷移した
  - (a) Cmd+N / Ctrl+N / New ボタンで新規ノートを作成
  - (b) 表示モードのカードをクリックして編集モードに切り替え
  - (c) 別カード編集中に新しいカードをクリック（自動保存後に新カードが EditMode）
  - (f) 検索結果・タグフィルタ適用後のカードクリック
- **期待結果**: CodeMirror 6 の選択位置がドキュメント末尾にある（`EditorView.state.selection.main.head === EditorView.state.doc.length`）
- **補足**: カーソル位置は EditMode を抜けるたびに揮発し、再遷移時は毎回 `doc.length` にリセットされる

#### AC-EDIT-14: Copy ボタンのコピー成功フィードバック

- **前提**: ノートカードが表示モードまたは編集モードである
- **操作**: Copy ボタンをクリックする、または `c` キーを押下する
- **期待結果**:

#### AC-EDIT-15: click-to-position の不実施

#### AC-NAV-10: カード間移動の境界動作

- **操作 B**: 最下部のカードで ↓ キーを押下する

#### AC-STOR-01: ファイル名規則 `YYYY-MM-DDTHHMMSS.md` [リリースブロッキング]

#### AC-STOR-02: 自動保存によるファイル永続化 [リリースブロッキング]

- **前提**: 編集モードでノートの内容を変更した
- **操作**: カード外をクリックして編集モードを終了する
- **期待結果**:
  1. 対応する `.md` ファイルの内容が変更後の内容と一致する
  2. ファイルの書き込みがアプリのクラッシュなしに完了する

#### AC-STOR-03: frontmatter 形式の正しさ [リリースブロッキング]

- **前提**: タグ `gpt`, `coding` を持つノートが保存される
- **期待結果**: ファイル先頭が以下の YAML frontmatter 形式である
  ```
  ---
  tags: [gpt, coding]
  ---
  ```
- **制約**: frontmatter に含まれるメタデータは `tags` のみ。作成日はファイル名から取得する。

#### AC-STOR-04: OS 別デフォルト保存ディレクトリ [リリースブロッキング]

- **前提**: 初回起動時に保存ディレクトリが未設定（`config.json` 不在）
- **期待結果**:
  - Linux: `~/.local/share/com.promptnotes/notes/`
  - macOS: `~/Library/Application Support/com.promptnotes/notes/`
- **制約**: デフォルトパスは ADR-010 で固定した Tauri bundle identifier `com.promptnotes` と `app_data_dir()` API から一意に導出される。`tauri.conf.json` の `identifier` を変更せず、`dirs::data_dir()` 等による OS 別パスのハードコードが行われていないこと。Rust 統合テストで `app_data_dir()` の戻り値が `<DATA>/com.promptnotes/` で終わることをアサートする

#### AC-STOR-05: 設定からの保存ディレクトリ変更 — 2 段階確定

- **前提**: 設定モーダルで保存ディレクトリの変更操作を行う
- **操作**:
  1. `[参照]` ボタンを押下し OS ネイティブダイアログで候補ディレクトリを選択
  2. 必要に応じて `[ ] 既存ノートを新ディレクトリへ移動する` チェックボックスを選択
  3. `[Apply]` ボタンを押下
  4. 移動オプションが選択されている場合、二次確認ダイアログ「元のディレクトリから削除されます。元に戻せません。実行しますか？」で `[実行]` を選択
- **期待結果**:
  1. `[参照]` 押下時点では `config.json` は書き換わらない（pending 状態）
  2. `[Apply]` 押下後、以降の新規ノートが新しいディレクトリに作成される
  3. 新しいディレクトリ内の既存 `.md` ファイルがフィードに表示される
  4. 移動オプションオフの場合、旧ディレクトリのファイルは一切変更されない（移動・削除・改変なし）
- **制約**: requirements 不可侵条項 1（2 段階確定）、条項 2（既存ノートの非改変原則）に準拠すること

#### AC-STOR-05a: 既存ノートの移動オプション

- **前提**: 旧 `notes_dir` に N 件の `.md` ファイルが存在し、新 `notes_dir` は空（またはファイル名衝突なし）
- **操作**: AC-STOR-05 の手順で、移動オプションを選択して Apply
- **期待結果**:
  1. 二次確認ダイアログが表示され、`[実行]` 押下後に 3 フェーズ（コピー → config atomic write → 旧削除）が実行される
  2. 新 `notes_dir` に N 件の `.md` ファイルが存在する
  3. 旧 `notes_dir` の `.md` ファイルは 0 件になる
  4. 旧 `notes_dir` の非 `.md` ファイル（画像・`.canvas` 等）は触られず残る
  5. `SetConfigResult.moved == N` かつ `SetConfigResult.remaining_in_old == 0`

#### AC-STOR-05b: 移動オプションでファイル名衝突検出

- **前提**: 旧 `notes_dir` と新 `notes_dir` に同名の `.md` ファイルが存在する
- **操作**: 移動オプションを選択して Apply
- **期待結果**:
  1. Phase 1 のコピー前に衝突が検出される
  2. `ConfigError::MoveConflict(filenames)` が返却される
  3. `config.json` は書き換わらない
  4. 旧 `notes_dir` と新 `notes_dir` の両方が `set_config` 呼び出し前と同一状態のまま（新側に途中コピーが残らない）
  5. UI が衝突ファイル名を表示し、手動解消を促すメッセージを表示する

#### AC-STOR-05c: 起動時ディレクトリ不在 — 自動フォールバック禁止

- **前提**: `config.json` の `notes_dir` が `/path/to/missing/dir` を指しており、該当ディレクトリが存在しない（または権限不足・外部デバイス切断・ディレクトリでない）
- **操作**: アプリを起動する
- **期待結果**:
  1. アプリは自動的にデフォルトディレクトリへフォールバックしない（`config.json` は書き換わらない）
  2. エラーモーダルが表示され、エラー種別に応じたメッセージが表示される:
     - `ENOENT` → 「保存ディレクトリが見つかりません」
     - `EACCES` → 「アクセス権限がありません」
     - `EIO`/`ENODEV`/`ESTALE` → 「外付けディスクまたはネットワークドライブが接続されていない可能性があります」
     - `ENOTDIR` → 「指定されたパスはディレクトリではありません」
  3. `[再試行] / [別のディレクトリを選ぶ] / [デフォルトに戻す]` の 3 択が表示される
  4. `[再試行]` はディレクトリ状態の再検証のみ行い（一時的不在からの復帰用）、成功すればそのまま起動継続
  5. `[デフォルトに戻す]` を選択した場合のみ、明示同意を経て `config.json` がデフォルト値で上書きされる
- **制約**: requirements 不可侵条項 4（起動時不在の自動フォールバック禁止）に準拠

#### AC-STOR-06: 本文往復冪等性（save / reopen で本文先頭が変化しない）[リリースブロッキング]

- **前提**: 任意の本文（例: `Hello` や先頭に日本語・空白・改行を含むテキスト）を持つノートが編集モードで開かれている
- **操作**: 編集内容を変更せずに編集モードを終了（カード外クリック等）して保存し、その後再度カードを開いて編集モードに遷移する
- **期待結果**:
  1. 再度開いた編集モードの本文が、保存前の本文と完全に一致する（先頭に改行 `\n` が追加されない）
  2. 上記の保存→再読込を任意の回数繰り返しても、本文先頭に改行が累積しない
- **制約**: ADR-008 で定めた body 意味論（body は frontmatter 閉じフェンスおよびその直後の区切り `\n` を含まない）に準拠し、Rust `storage/frontmatter.rs` および TypeScript `src/editor/frontmatter.ts` / `tests/unit/frontmatter.ts` の両側で `parse → serialize → parse` の往復冪等性が成立すること。

## 3. Failure Criteria

以下の条件のいずれかに合致した場合、リリースを不可とする。

### FC-SCOPE-01: タイトル入力欄の存在 [リリースブロッキング]

- **条件**: 新規ノート作成画面または編集モードにタイトル入力欄（`<input>` や専用テキストフィールド）が存在する
- **根拠**: Convention 2 — タイトル入力欄禁止はスコープ外として明示されており、実装した場合リリース不可
- **検出方法**: DOM 内にノートタイトル用の入力要素が存在しないことをアサーション

### FC-SCOPE-02: Markdown プレビュー（レンダリング）の存在 [リリースブロッキング]

- **条件**: Markdown を HTML にレンダリングして表示するプレビューパネルまたはモードが存在する
- **根拠**: Convention 2 — Markdownプレビュー（レンダリング）禁止
- **検出方法**: 編集モード中に `.cm-editor` 以外の Markdown レンダリング出力（`<h1>`, `<strong>` など Markdown 由来の HTML タグ）が表示されないことをアサーション

### FC-SCOPE-03: AI 呼び出し機能の存在

- **条件**: アプリ内から外部 AI API を呼び出す機能（ボタン、メニュー、自動トリガー）が存在する
- **検出方法**: ネットワークリクエストの監視で AI API エンドポイントへの通信がないこと

### FC-STOR-01: ファイル名規則違反 [リリースブロッキング]

- **条件**: 生成されたノートファイルの名前が `YYYY-MM-DDTHHMMSS.md` 形式に合致しない
- **検出方法**: 保存ディレクトリ内の全ファイル名を正規表現 `^\d{4}-\d{2}-\d{2}T\d{6}\.md$` で検証

### FC-STOR-02: 自動保存失敗 [リリースブロッキング]

- **条件**: 編集モード終了後にファイルが更新されない、または編集内容とファイル内容が不一致
- **検出方法**: 編集→表示モード遷移後にファイル読み込みで内容を比較

### FC-EDIT-01: Cmd+N / Ctrl+N 未応答 [リリースブロッキング]

- **条件**: キーボードショートカット Cmd+N (macOS) / Ctrl+N (Linux) を押下しても新規ノートが作成されない
- **検出方法**:
  1. ショートカット押下後にフィード最上部に編集モードのカードが存在しないことで判定
  2. **分離検証（必須）**: `+ New Note` ボタンを介さず、ショートカット単独で保存ディレクトリ内の `.md` ファイル件数が 0 → 1 に増えることを確認する。ボタン動作と相乗させた検証（例: `navigateToView('editor')` 後に Ctrl+N を打って `>= 1` をアサートする形）はボタン由来の作成でショートカット障害を覆い隠すため不可とする

### FC-EDIT-02: Copy ボタン未動作 [リリースブロッキング]

- **条件**: Copy ボタンクリック後にクリップボードにノート本文がコピーされない
- **検出方法**: Clipboard API の読み取りでコピー内容を検証

### FC-EDIT-03: Copy ボタンのモード別欠落 [リリースブロッキング]

- **条件**: Copy ボタンが表示モードまたは編集モードのいずれか一方でしか表示されない、あるいはモード遷移時に一時的にでも描画されない瞬間が存在する
- **根拠**: requirements「表示モード・編集モードのどちらでも常時表示」および AC-EDIT-06 / AC-EDIT-06b
- **検出方法**: ノートカードを表示モード → 編集モード → 表示モードと遷移させる過程で、Copy ボタンの DOM 要素 (`aria-label="本文をコピー"` 等の識別子) が各ステップで常に存在しクリック可能であることをアサーション

### FC-FEED-01: デフォルトフィルタ欠如 [リリースブロッキング]

- **条件**: アプリ起動直後に全期間のノートが表示される（直近7日間フィルタが適用されていない）
- **検出方法**: 7日より前のタイムスタンプを持つテストノートが初期表示に含まれていないことをアサーション

### FC-FEED-02: 検索・フィルタ未実装 [リリースブロッキング]

- **条件**: 全文検索バー、タグフィルタ、日付フィルタのいずれかが機能しない
- **検出方法**: 各フィルタ操作後の表示ノート数とフィルタ条件の整合性を検証

### FC-EDITOR-01: CodeMirror 6 未使用 [リリースブロッキング]

- **条件**: 編集モードで CodeMirror 6 インスタンス（`.cm-editor` DOM 要素）が検出されない
- **根拠**: Convention 2 — CodeMirror 6 必須

### FC-UI-01: カード内スクロールの存在

- **条件**: ノートカード内にスクロールバーが表示される、または `overflow: scroll` / `overflow: auto` でコンテンツがクリップされている
- **検出方法**: カードコンテナの `overflow` CSS プロパティと `scrollHeight > clientHeight` の検証

### FC-UI-02: デフォルトテーマでのノート間罫線の存在

- **条件**: デフォルトテーマでノートカード間に `border`, `<hr>`, divider 等の罫線が存在する
- **検出方法**: ノートカード間の DOM 要素の `border` CSS プロパティおよび `<hr>` 要素の不在を検証

## 4. E2E Test Generation Meta-Prompt

### 目的

本セクションは `codd propagate` による E2E テスト自動生成の機械可読命令である。受入基準（セクション2）および失敗基準（セクション3）からテストシナリオを導出し、Playwright を用いて Tauri アプリケーションの E2E テストを生成する。

### テストランタイム環境

#### アプリケーション起動シーケンス

1. `npm run build` または `pnpm build` で Svelte フロントエンドをビルドする
2. `cargo tauri dev` で Tauri 開発サーバーを起動する（テスト時はヘッドレス WebView）
3. ヘルスチェック: WebView が ready 状態になるまで最大 30 秒待機する
4. CI 環境では `xvfb-run` (Linux) を使用して仮想ディスプレイ上で実行する

#### ブラウザ設定

```typescript
{
  headless: true,
  timeout: 30000,
  viewport: { width: 1280, height: 720 }
}
```

Tauri アプリのため、Playwright の Electron 統合モード (`_electron`) または WebDriver 経由で WebView に接続する。`package.json` の `scripts` および `tauri.conf.json` からエントリポイントを自動検出すること。

### MECE ドメイン分割

| ドメイン | 責務 | 対応受入基準 |
|---|---|---|
| `editor` | ノート作成・編集・表示モード遷移・CodeMirror 6・Copy フィードバック・click-to-position 不実施 | AC-EDIT-01〜15, FC-EDIT-01, FC-EDIT-02, FC-EDIT-03, FC-EDITOR-01 |
| `keyboard-nav` | カードフォーカス状態・↑/↓ 移動・Enter/Esc/c/d キー操作・矢印キー有効範囲 | AC-NAV-01〜10 |
| `storage` | ファイル名規則・自動保存・frontmatter・ディレクトリ・本文往復冪等性 | AC-STOR-01〜06, FC-STOR-01, FC-STOR-02 |
| `feed` | フィード表示・フィルタ・検索・ソート | AC-FEED-01〜05, FC-FEED-01, FC-FEED-02 |
| `scope-guard` | スコープ外機能が存在しないことの検証 | FC-SCOPE-01, FC-SCOPE-02, FC-SCOPE-03 |
| `settings` | 設定モーダル・ディレクトリ変更 | AC-UI-01, AC-STOR-05 |
| `ui-layout` | ヘッダー構成・アプリ名非表示・カード表示・カード高さ・ノート間区切り・テーマ・視認性・アニメーション | AC-UI-02〜10, FC-UI-01, FC-UI-02 |

### テストレベル分離

#### API 統合テスト（Tauri コマンドレベル）

PromptNotes は Web アプリケーションではなく Tauri デスクトップアプリのため、API 統合テストは HTTP エンドポイントではなく Tauri IPC コマンド（`invoke`）を対象とする。

- Tauri の `invoke` コマンド呼び出しによるファイル CRUD 操作のレスポンスを検証する
- ファイル名規則、frontmatter 形式、保存ディレクトリパスなどのデータ契約を検証する
- ファイルシステムの状態（ファイル存在、内容一致）を直接アサーションする
- サーバーヘルスベースライン: 各 IPC 呼び出しがエラー（パニック、unhandled exception）を返さないことを、ビジネスロジックのアサーションに先立って検証する。IPC エラーは 5xx 相当のサーバーエラーであり、ビジネスロジックの成否とは分類が異なる

ファイル命名規則: `tests/e2e/<domain>.spec.ts`

#### ブラウザテスト（WebView 操作レベル）

- Tauri WebView 内の実際の DOM を操作する：ボタンクリック、テキスト入力、キーボードショートカット押下
- 各ユーザー操作後、結果として表示される UI 状態（DOM 要素の存在・内容・スタイル）をアサーションする
- ページ遷移を伴う操作（設定モーダル開閉など）では、遷移後の表示要素を必ず検証する。HTTP ステータスのみの確認は不十分とする — 200 でも表示内容が誤っている場合やサイレントリダイレクトを検出するため、URL アサーションと可視コンテンツ要素の両方を検証する
- サーバーヘルスベースライン: Tauri バックエンドへの IPC 呼び出しが例外を返さないことを、ビジネスロジックのアサーションに先立って検証する

ファイル命名規則: `tests/e2e/<domain>.browser.spec.ts`

### シナリオ導出ルール

1. **正常系**: 各受入基準 (AC-*) の「期待結果」を1つ以上のテストケースに変換する。複数ステップの状態遷移（例: 表示モード → クリック → 編集モード → Esc → カードフォーカス状態 → Esc → フォーカスなし状態）はチェーン全体の各リンクを個別にアサーションする
2. **異常系**: 各失敗基準 (FC-*) を反転し、禁止状態が発生しないことをアサーションするテストケースとする
3. **境界値**: AC-FEED-01 の7日間境界、AC-STOR-01 のタイムスタンプ秒精度、AC-NAV-10 の最上部/最下部カードでの境界動作を境界値テストとして追加する
4. **排他制御**: AC-EDIT-10 の同時編集排他は、連続するカード切り替え操作で検証する
5. **キーボードナビゲーション**: AC-NAV-01〜10 のフォーカス状態遷移チェーン（編集モード → Esc → カードフォーカス → ↑/↓ 移動 → Enter → 編集モード）を一連のシーケンスとして検証する。矢印キーの有効範囲制限（AC-NAV-09）は編集モード中・検索バーフォーカス中の両コンテキストで個別にテストする
6. **アニメーション**: AC-UI-09 のフェードイン/フェードアウト/FLIP アニメーションは、DOM 要素の `opacity` や `transform` CSS プロパティの変化を検証する。AC-UI-10 の操作非ブロックは、アニメーション実行中に連続削除操作を行い、全操作が正しく完了することを検証する。AC-UI-11 のモード切替高さアニメーションは、モード切替前後でカード要素の高さが変化していること（または CSS `transition` が設定されていること）を検証する。AC-UI-12 の prefers-reduced-motion 対応は、メディアクエリを強制的に有効化した状態でアニメーションが無効になることを検証する

### アーキテクチャ適応ルール

- テスト生成時に `src-tauri/src/` 内の Tauri コマンド定義と `src/` 内の Svelte コンポーネント構造をスキャンする
- 未実装のコマンドまたはコンポーネントに依存するテストは `test.fixme()` でマークし、スキップではなく明示的に未実装であることを記録する
- `tauri.conf.json` の `allowlist` からアプリが使用する Tauri API を検出し、テストのセットアップに反映する
