# Acceptance Criteria

## 1. Overview

PromptNotes は、AI へ渡すプロンプトを素早く書き溜めるためのローカルノートアプリである。Tauri（Rust + WebView）と Svelte で構築し、CodeMirror 6 エディタでMarkdownシンタックスハイライト付きのプレーンテキスト編集を提供する。データはローカル `.md` ファイルとして保存し、クラウド同期・AI呼び出し機能は持たない。

本ドキュメントは、PromptNotes のリリース判定に使用する受け入れ基準・不合格基準を定義し、E2E テスト自動生成のためのメタプロンプトを提供する。

**対象プラットフォーム:** Linux, macOS（Windows は将来対応のためスコープ外）

### Non-negotiable conventions への適合

| Convention | 対象モジュール | 適合方針 |
|---|---|---|
| Cmd+N / Ctrl+N 即時新規ノート作成・1クリックコピーボタン | `module:editor` | AC-ED-04, AC-ED-05 で検証。未実装はリリース不可。 |
| 新規ノート編集中でも Cmd+N / Ctrl+N で追加の新規ノートを作成可能 | `module:editor` | AC-ED-08 で検証。未実装はリリース不可。 |
| エディタ画面に過去のノートのインラインリスト表示・選択でその場編集 | `module:editor` | AC-ED-07 で検証。未実装はリリース不可。 |
| CodeMirror 6 必須・タイトル入力欄禁止・Markdownプレビュー禁止 | `module:editor` | AC-ED-01, FC-ED-01, FC-ED-02 で検証。違反はリリース不可。 |
| ファイル名規則 `YYYY-MM-DDTHHMMSS.md`・自動保存 | `module:storage` | AC-ST-01, AC-ST-02, FC-ST-01 で検証。違反はリリース不可。 |
| デフォルト直近7日間フィルタ・タグ/日付フィルタ・全文検索 | `module:grid` | AC-GR-01〜AC-GR-04 で検証。未実装はリリース不可。 |

## 2. Acceptance Criteria

### エディタ画面（module:editor）

| ID | 基準 | 検証方法 |
|---|---|---|
| AC-ED-01 | エディタは CodeMirror 6 で実装されており、Markdown シンタックスハイライトが有効である（レンダリングではなくプレーンテキスト表示）。 | ブラウザテスト：エディタ領域に Markdown 記法（`# heading`, `**bold**`）を入力し、対応するハイライト用 CSS クラスが DOM に付与されていることを確認。レンダリング済み HTML 要素（`<h1>`, `<strong>` 等）が本文領域に存在しないことを確認。 |
| AC-ED-02 | タイトル入力欄が存在しない。画面上に本文エディタ領域と frontmatter 領域のみが表示される。 | ブラウザテスト：エディタ画面のDOMにタイトル専用の `input` / `textarea` が存在しないことを確認。 |
| AC-ED-03 | 画面上部に frontmatter（タグ入力）領域があり、背景色で本文と視覚的に区別される。frontmatter は YAML 形式で `tags` フィールドのみ保持する。 | ブラウザテスト：frontmatter 領域の `background-color` が本文領域と異なることを確認。保存されたファイルの frontmatter が `tags` キーのみを含む YAML であることをファイル読み取りで検証。 |
| AC-ED-04 | **Cmd+N**（macOS）/ **Ctrl+N**（Linux）を押すと即座に新規ノートが作成され、エディタにフォーカスが移動する。 | ブラウザテスト：キーボードショートカットを送信し、新しい空のエディタ画面が表示されること、対応する `.md` ファイルがストレージディレクトリに作成されることを確認。フォーカスがエディタ領域にあることを `document.activeElement` で検証。 |
| AC-ED-05 | **1クリックコピーボタン**が表示されており、クリックすると本文全体（frontmatter を除く）がクリップボードにコピーされる。 | ブラウザテスト：本文を入力後、コピーボタンをクリックし、クリップボード API の内容が本文と一致することを確認。 |
| AC-ED-06 | 保存は自動で行われ、ユーザーによる明示的な保存操作は不要である。 | ブラウザテスト：本文を入力後、一定時間待機し、対応するファイルの内容が入力内容と一致することをファイルシステム経由で検証。 |
| AC-ED-07 | エディタ画面は、現在編集中のノートに加え、過去のノートをスクロール可能なリストとして表示する。各ノートは frontmatter（タグ入力）領域と本文領域を持ち、任意のノート本文をクリック／フォーカスするとその場で編集できる。 | ブラウザテスト：複数の既存ノートを作成した状態でエディタ画面を開き、複数のノートブロックが DOM 上に同時に描画されていることを確認。任意の過去ノート本文にキー入力を行い、対応する `.md` ファイルが `save_note` 経由で更新されることをファイルシステム経由で検証。 |
| AC-ED-08 | **新規ノートを編集している状態**であっても、**Cmd+N**（macOS）/ **Ctrl+N**（Linux）を押すと即座にさらに新しいノートが作成され、新規ノートにフォーカスが移動する。既存の新規ノートは失われない。 | ブラウザテスト：`Cmd+N` / `Ctrl+N` で新規ノートを作成 → 本文を数文字入力（未保存状態を含む）→ 再度 `Cmd+N` / `Ctrl+N` を送信し、追加の新規 `.md` ファイルがストレージディレクトリに作成され、フォーカスが新しいノートに移動することを確認。先に入力した未保存内容が後続のデバウンス完了後に元のファイルへ保存されることを検証。 |

### ストレージ（module:storage）

| ID | 基準 | 検証方法 |
|---|---|---|
| AC-ST-01 | ノート作成時のファイル名が `YYYY-MM-DDTHHMMSS.md` 形式（例: `2026-04-04T143205.md`）である。タイムスタンプはノート作成時刻で確定する。 | API テスト：新規ノートを作成し、ストレージディレクトリ内のファイル名が正規表現 `^\d{4}-\d{2}-\d{2}T\d{6}\.md$` に合致することを確認。 |
| AC-ST-02 | ファイル構造は YAML frontmatter（`tags` のみ）＋本文で構成される。 | API テスト：保存済みファイルを読み取り、`---` で囲まれた YAML frontmatter ブロックが先頭にあり、`tags` キーが配列値を持つことをパースして検証。 |
| AC-ST-03 | デフォルト保存ディレクトリは Linux: `~/.local/share/promptnotes/notes/`、macOS: `~/Library/Application Support/promptnotes/notes/` である。 | API テスト：初回起動時にデフォルトディレクトリが使用されていることを設定値またはファイルパスから検証。 |
| AC-ST-04 | 設定画面から保存ディレクトリを任意のパスに変更でき、変更後は新規ノートが新ディレクトリに作成される。 | ブラウザテスト：設定画面でディレクトリを変更後、新規ノートを作成し、指定ディレクトリ内にファイルが存在することを確認。 |

### グリッドビュー（module:grid）

| ID | 基準 | 検証方法 |
|---|---|---|
| AC-GR-01 | グリッドビューは Pinterest スタイルの可変高カードでノート一覧を表示する。 | ブラウザテスト：グリッドビューを開き、カード要素が複数カラムで可変高さのレイアウトで描画されていることを DOM / CSS で検証。 |
| AC-GR-02 | デフォルトフィルタとして**直近7日間**のノートのみが表示される。7日より前のノートは初期状態では非表示である。 | ブラウザテスト：8日以上前のタイムスタンプを持つファイルと当日のファイルを作成し、グリッドビューを開いた際に当日のノートのみが表示され、古いノートが表示されないことを確認。 |
| AC-GR-03 | タグおよび日付でフィルタリングが可能である。フィルタ適用後、条件に合致するノートのみが表示される。 | ブラウザテスト：異なるタグ・日付のノートを複数作成し、タグフィルタ・日付フィルタを操作して表示結果が正しく絞り込まれることを確認。 |
| AC-GR-04 | 全文検索（ファイル全走査）が機能し、検索クエリに合致する本文を含むノートのみが表示される。 | ブラウザテスト：一意の文字列を含むノートを作成し、検索ボックスにその文字列を入力して該当ノートのみが表示されることを確認。 |
| AC-GR-05 | カードをクリックするとエディタ画面に遷移し、該当ノートの内容が表示される。 | ブラウザテスト：グリッドビューでカードをクリックし、エディタ画面に遷移して正しい本文が表示されることを確認。 |

### 設定画面

| ID | 基準 | 検証方法 |
|---|---|---|
| AC-CF-01 | 設定画面で保存ディレクトリの変更が可能である。 | ブラウザテスト：設定画面を開き、ディレクトリパスを入力・保存し、設定が永続化されることを確認。 |

### 配布・ビルド

| ID | 基準 | 検証方法 |
|---|---|---|
| AC-DI-01 | Linux 向けにバイナリ直接ダウンロード、Flatpak（Flathub）、NixOS パッケージが提供される。 | CI パイプラインでビルド成果物が生成されることを確認。 |
| AC-DI-02 | macOS 向けにバイナリ直接ダウンロード、Homebrew Cask が提供される。 | CI パイプラインでビルド成果物が生成されることを確認。 |

## 3. Failure Criteria

以下のいずれかに該当する場合、リリース不可とする。

### エディタ画面（module:editor）— リリースブロッカー

| ID | 不合格条件 |
|---|---|
| FC-ED-01 | エディタが CodeMirror 6 以外のエンジン（Monaco, textarea, contenteditable 等）で実装されている。 |
| FC-ED-02 | タイトル入力欄が存在する、または Markdown プレビュー（レンダリング）機能が実装されている。これらはスコープ外として明示的に除外されており、存在自体がリリース不可要因である。 |
| FC-ED-03 | Cmd+N / Ctrl+N で新規ノートが作成されない、またはフォーカスがエディタに移動しない。 |
| FC-ED-04 | 1クリックコピーボタンが存在しない、またはクリックしても本文がクリップボードにコピーされない。 |
| FC-ED-05 | frontmatter 領域が背景色で本文と視覚的に区別されていない。 |
| FC-ED-06 | エディタ画面に過去のノートがリスト表示されない、またはリスト上のノートをクリックしてもその場で編集できない（別画面へ遷移する、読み取り専用になる等）。 |
| FC-ED-07 | 新規ノートを編集中の状態で Cmd+N / Ctrl+N を押しても追加の新規ノートが作成されない、またはフォーカスが移動しない。 |

### ストレージ（module:storage）— リリースブロッカー

| ID | 不合格条件 |
|---|---|
| FC-ST-01 | ファイル名が `YYYY-MM-DDTHHMMSS.md` 形式に従っていない（例: UUID、連番、タイトルベースのファイル名）。 |
| FC-ST-02 | 自動保存が機能せず、ユーザーが手動で保存操作を行わなければデータが永続化されない。 |
| FC-ST-03 | frontmatter に `tags` 以外のメタデータフィールドが自動挿入される。 |

### グリッドビュー（module:grid）— リリースブロッカー

| ID | 不合格条件 |
|---|---|
| FC-GR-01 | グリッドビューのデフォルト表示が直近7日間にフィルタされていない（全件表示やフィルタなし）。 |
| FC-GR-02 | タグフィルタまたは日付フィルタが未実装である。 |
| FC-GR-03 | 全文検索が未実装である。 |
| FC-GR-04 | カードクリックでエディタ画面に遷移しない。 |

### スコープ外機能の混入 — リリースブロッカー

| ID | 不合格条件 |
|---|---|
| FC-SC-01 | AI 呼び出し機能が実装されている。 |
| FC-SC-02 | クラウド同期機能が実装されている。 |
| FC-SC-03 | モバイル対応が含まれている。 |

### 一般

| ID | 不合格条件 |
|---|---|
| FC-GN-01 | Linux または macOS でアプリケーションが起動しない。 |

## 4. E2E Test Generation Meta-Prompt

以下は `codd propagate` が E2E テストを自動生成するための機械可読メタプロンプトである。

### アプリケーション起動シーケンス

PromptNotes は Tauri デスクトップアプリケーションであるため、E2E テスト実行時には以下の手順でアプリケーションを起動する。

1. `package.json` の `scripts` セクションから開発サーバー起動コマンドを検出する（例: `tauri dev`, `npm run dev`）。
2. 開発サーバーをバックグラウンドで起動し、WebView が HTTP でアクセス可能になるまで待機する（デフォルト: `http://localhost:1420`）。
3. ヘルスチェック: HTTP GET でフロントエンドが 200 を返すことを確認してからテスト実行を開始する。
4. CI 環境では `headless: true` でブラウザを起動する。ローカル開発時は `headless: false` を許容する。

### テストレベル分離

E2E テストは以下の2レベルに厳密に分離する。

| レベル | 目的 | ツール | ファイル命名規則 |
|---|---|---|---|
| API インテグレーションテスト | Tauri コマンド（IPC）およびファイルシステム操作の検証。ステータスコード・データ契約を確認。 | Playwright `request` context または Tauri テストユーティリティ | `tests/e2e/<domain>.spec.ts` |
| ブラウザテスト | 実際のユーザー操作シミュレーション。クリック・キー入力・画面遷移・表示内容を検証。 | Playwright `page`（WebView 接続） | `tests/e2e/<domain>.browser.spec.ts` |

**ブラウザテスト必須ルール:**

- 画面遷移を伴うユーザー操作（カードクリック→エディタ遷移、Cmd+N→新規ノート画面）では、遷移後の URL アサーションと遷移先ページの可視コンテンツ要素のアサーションを必ず両方行う。HTTP ステータスのみの確認は不十分であり、200 で誤ったコンテンツが返るケースを検出できない。
- PromptNotes はローカルデスクトップアプリのため認証フローは不要だが、初回起動時のデフォルトディレクトリ作成確認フローをベースラインテストとして含める。

**サーバーヘルスベースライン:**

- すべての HTTP リクエストアサーションは、ビジネスロジックのステータスコード（200, 404 等）を検証する前に、レスポンスステータスが 500 未満であることを最初に確認する。5xx はサーバーエラー（未処理例外、DB停止等）であり、4xx（認証失敗、未検出等）とは本質的に異なる。

### MECE ドメイン分割

| ドメイン | 対象範囲 | API テストファイル | ブラウザテストファイル |
|---|---|---|---|

| `storage` | ファイル名規則 `YYYY-MM-DDTHHMMSS.md`、ファイル構造（YAML frontmatter + 本文）、デフォルトディレクトリ、ディレクトリ変更 | `tests/e2e/storage.spec.ts` | `tests/e2e/storage.browser.spec.ts` |
| `grid` | グリッドビュー表示、デフォルト7日間フィルタ、タグ/日付フィルタ、全文検索、カードクリック遷移 | `tests/e2e/grid.spec.ts` | `tests/e2e/grid.browser.spec.ts` |

### シナリオ導出ルール

1. **ポジティブシナリオ:** 各受け入れ基準（AC-*）から正常系テストシナリオを1つ以上導出する。
2. **ネガティブシナリオ:** 各不合格基準（FC-*）を反転させ、「その状態が存在しないこと」をアサーションとするテストシナリオを導出する。
3. **境界値シナリオ:** デフォルト7日間フィルタでは、ちょうど7日前（含まれる）と8日前（含まれない）の境界をテストする。

### アーキテクチャ適応ルール

- テスト生成時に実際のルート/エンドポイント構造（Svelte のルーティング定義または Tauri コマンド定義）をスキャンする。
- 受け入れ基準に対応するルート/コマンドが未実装の場合、`test.skip()` ではなく `test.fixme()` でマークし、未実装であることを明示する。`test.skip()` は使用禁止。
- Tauri IPC コマンド名は `src-tauri/src/` 配下の `#[tauri::command]` アノテーションからスキャンする。
- Svelte ルートは `src/routes/` または `src/lib/` 配下のファイル構造からスキャンする。

### 品質ゲート

| 条件 | 基準 |
|---|---|
| テスト結果 | 全テスト PASS |
| SKIP 数 | 0（`test.fixme()` は許容するが `test.skip()` は禁止） |
| 受け入れ基準カバレッジ | AC-* 基準すべてに対応するテストが1つ以上存在する |
| 不合格基準カバレッジ | FC-* 基準すべてに対応するガードテストが1つ以上存在する |
| リリースブロッカー | `module:editor` の Cmd+N・コピーボタン、CodeMirror 6 使用・タイトル欄禁止・プレビュー禁止、`module:storage` のファイル名規則・自動保存、`module:grid` の7日間フィルタ・タグ/日付フィルタ・全文検索 — これらに対応するテストが1つでも FAIL なら CI パイプラインを失敗とする |

### 共有ヘルパーディレクトリ

`tests/e2e/helpers/` に以下のヘルパーモジュールを配置し、テストファイル間の重複を排除する。

| ファイル | 責務 |
|---|---|
| `tests/e2e/helpers/app-launch.ts` | Tauri 開発サーバー起動・WebView 接続・ヘルスチェック待機 |
| `tests/e2e/helpers/test-data.ts` | テスト用 `.md` ファイルの作成・削除、frontmatter 生成、タイムスタンプ付きファイル名生成 |
| `tests/e2e/helpers/assertions.ts` | 共通アサーション（ファイル名形式検証、frontmatter パース検証、5xx ガードアサーション） |
| `tests/e2e/helpers/editor.ts` | エディタ操作ヘルパー（CodeMirror への入力、コピーボタンクリック、Cmd+N 送信） |
| `tests/e2e/helpers/grid.ts` | グリッドビュー操作ヘルパー（フィルタ操作、検索入力、カード選択） |
