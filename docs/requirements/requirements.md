# PromptNotes — Requirements

## コンセプト

AIへ渡すプロンプトを素早く書き溜めるためのノートアプリ。
タイトル不要、本文だけ。書いたらすぐ次へ。すべてが1画面で完結する。

ターミナルやIDEにプロンプトをペーストする用途を想定しており、AI呼び出し機能は持たない。

---

## 技術スタック

| 項目 | 決定 |
|---|---|
| フレームワーク | Tauri（Rust + WebView） |
| フロントエンド | Svelte |
| エディタ | CodeMirror 6 |
| データ保存 | ローカル .md ファイル |
| 検索 | ファイル全走査（件数が少ないため十分） |
| ターゲット | Linux, macOS（Windows は将来対応） |

React vs Svelte: コンパイルされるsvelteを採用する。

### Tauriを選んだ理由

- テキストエディタはIME・カーソル・コピー操作など入力周りが複雑で、WebViewに任せた方が罠が少ない
- ElectronよりバイナリサイズとメモリフットプリントがGhostty方式（Zig + Native UI）の二重UI実装よりも現実的
- Flatpak / Homebrew との相性が良い

### CodeMirrorを選んだ理由

- バンドルサイズが軽量（Monaco比で1/10以下）
- Obsidianと同じエンジンでユーザーの指の慣れと一致
- frontmatter領域だけ背景色を変えるなどのカスタマイズがしやすい
- 公式のMarkdownシンタックスハイライトパッケージあり

### .mdファイルを選んだ理由

- ObsidianやVSCodeなど他ツールでそのまま開ける
- Obsidian vault内の1ディレクトリをこのアプリに割り当てる使い方ができる
- Gitで自然にバージョン管理できる
- 検索速度の遅さは問題にならない（1週間分＝数十件程度）

## データ設計

### ファイル名規則

```
YYYY-MM-DDTHHMMSS.md
例: 2026-04-04T143205.md
```

タイムスタンプのみ。タイトルは不要。

### ファイル構造

```
---
tags: [gpt, coding]
---

本文をここに書く...
```

- frontmatterはYAML形式
- メタデータはtagsのみ（作成日はファイル名から取得）

### デフォルト保存ディレクトリ

```
Linux:   ~/.local/share/com.promptnotes/notes/
macOS:   ~/Library/Application Support/com.promptnotes/notes/
```

デフォルトパスは Tauri bundle identifier `com.promptnotes`（ADR-010 で固定）と `app_data_dir()` から導出される。identifier・`app_data_dir()` 経由のパス解決・上記リテラル表記は単一の決定として整合している必要があり、いずれか 1 つを変更する場合は 3 つすべてを同時に更新すること。

### 保存ディレクトリの変更

「設定」から任意のディレクトリに変更可能。Obsidian vault内のサブディレクトリを指定する使い方を想定。以下を**不可侵条項**として定める。

1. **2 段階確定**: ディレクトリ選択は「参照で候補を選ぶ（pending）」→「Apply で確定」の 2 ステップに分離する。参照した瞬間に設定を書き換えない。
2. **既存ノートの非改変原則**: 保存ディレクトリの変更に伴ってユーザーのノートファイルを自動で移動・削除・改変してはならない。既定は「旧ディレクトリをそのまま残し、以降の新規ノートは新ディレクトリへ保存する」。
3. **移動は明示同意必須**: 旧ディレクトリから新ディレクトリへの既存ノート移動は、ユーザーがチェックボックスで選択し、かつ二次確認ダイアログ「元のディレクトリから削除されます。元に戻せません。実行しますか？」に同意した場合に限り実行する。非 `.md` ファイルは移動対象外とし、ユーザーの他資産（Obsidian の添付画像・`.canvas` 等）を破壊しない。
4. **起動時不在の自動フォールバック禁止**: アプリ起動時に設定済みディレクトリにアクセスできない場合、デフォルトへ自動的にフォールバックしない。エラー種別（削除・権限不足・外部デバイス切断・ディレクトリでない）に応じたメッセージを表示し、ユーザーに `[再試行] / [別のディレクトリを選ぶ] / [デフォルトに戻す]` の 3 択を提示する。一時的不在（外付けディスク未接続、NFS 切断）で設定が勝手に失われることを防ぐ。

## 機能要件

### トップフィード画面

アプリ起動時に表示される唯一のメイン画面。ノートが新しい順に縦に並ぶフィード形式。

ワイヤーフレーム: [top-feed.drawio](wire-frames/top-feed.drawio)

#### ヘッダー

- アプリ名は表示しない（画面領域をノートに最大限使う）
- 検索バー（全文検索、ファイル全走査）
- タグ・日付フィルタ
- **New** ボタン — 新規ノート作成（Cmd+N / Ctrl+N と同等）
- **⚙️** ボタン — 設定モーダルを開く

#### ノート

- ノートカードの高さは表示モードと編集モードで挙動が異なる：
  - **表示モード** — カードに最大高さ（max-height）を設け、本文がそれを超える場合はカード内スクロールで本文を表示する。max-height のデフォルト値は **300px**。当面はアプリケーション内の定数として保持する（設定モーダルからの変更および `config.json` への永続化は本リリースのスコープ外。将来 config に昇格させる余地を残す）
  - **編集モード** — カード内スクロールは設けず、本文全量に応じてカードの高さを伸ばし、全文をスクロールなしで表示する
  - 「カード外形高さ」を以下のように定義する（モード切替アニメーションの対象）：
    - 表示モード = `min(本文全展開高さ + パディング, max-height)`
    - 編集モード = `本文全展開高さ + パディング`
  - 表示モードでカード内スクロール位置を持っていた場合、編集モードへの遷移時にスクロール位置は引き継がず、エディタはノート先頭から表示する。表示モードに戻ったときも、スクロール位置はリセット（最上部）してよい
- デフォルトは**表示モード**（本文プレーンテキスト + タグ + タイムスタンプ）
- ノートをクリックで**編集モード**に切り替え（CodeMirror 6 が出現）
  - Markdownシンタックスハイライト（レンダリングなし）
  - 上部にfrontmatter（タグ入力）領域、背景色で視覚的に区別
  - **Undo / Redo 対応必須** — 編集モード中に **Cmd+Z / Ctrl+Z で元に戻し、Cmd+Shift+Z / Ctrl+Y でやり直し**。CodeMirror 6 の履歴機能を正規の仕組みとする（ブラウザ標準の contentEditable Undo には依存しない）
- 別のノートをクリック or 外部クリック or **Esc キー**で自動保存し表示モードに戻る
- 同時に編集モードになるノートは1つだけ
- **モード切替アニメーション** — 表示モード↔編集モードの切替時に、**カード外形高さ**（上記の定義に従う）の変化を 200ms / ease-in-out でアニメーションする：
  - 両方向（表示→編集・編集→表示）に適用する
  - 本文が短く表示モードと編集モードでカード外形高さが等しい場合、高さアニメーションは実質発火しない（差が 0）。これは正常動作として扱い、コンテンツ（本文テキスト / CodeMirror）の切替は他ケースと同様に即時行う
  - タイピング中のリアルタイムな高さ変化にはアニメーションを適用しない
  - 高さアニメーションの開始と同時にコンテンツ（本文テキスト / CodeMirror）は即時切替する
  - アニメーション中に別の操作（他ノートのクリック等）が発生した場合は即座に最終状態にスナップし、次の操作を受け付ける
  - `prefers-reduced-motion: reduce` が有効な環境ではアニメーションを無効化して即時切替にフォールバックする
- **1クリックCopyボタン** — 本文全体をクリップボードにコピー（核心UX）。**表示モード・編集モードのどちらでも常時表示**し、モード切替によってボタンが消えないこと。編集モード時は CodeMirror の本文入力操作と競合しない位置に配置する。コピー成功時は**ボタンの見た目が一瞬変わる**（チェックマーク化・色変化等）ことでフィードバックする。キーボードショートカット `c` によるコピー時も同じフィードバックを表示する。
- **Deleteボタン** — ノートを削除（確認ダイアログなし、即時削除）

#### 新規ノート作成

- **Cmd+N / Ctrl+N** またはNewボタンで即座にフィード最上部に新規ノートを編集モードで挿入
- フォーカスは自動的に新規ノートの本文へ移動

#### カーソル初期位置

- 編集モードに遷移したとき、カーソルは常に**本文（Body）の末尾**に置く
  - 新規ノート（Body 空）・既存ノートを問わず同一ルール
  - カーソル位置の永続化は行わない（編集モードを抜けるたびにリセット）
- クリック位置へのカーソル配置（click-to-position）は行わない

#### キーボードナビゲーション

##### カードフォーカス状態

ノートカードがフォーカスされている状態。フォーカス中のカードは **outline と box-shadow による強調**で視覚的に示す。

- **Esc キーで編集モードを抜ける** — 自動保存して表示モードに戻り、そのカードにフォーカスが残る
- **カードフォーカス状態で Esc** — フォーカスが外れる（どこにもフォーカスがない状態）
- **↑ / ↓ キー** — 隣のカードにフォーカスを移動
  - 最上部で ↑ → 何も起きない
  - 最下部で ↓ → より古いノートをロードする（スクロール時の追加ロードと同じ挙動）。古いノートがなければ何も起きない
- **Enter キー** — フォーカス中のカードが編集モードに入る
- **`c` キー** — 本文全体をクリップボードにコピー（Copy ボタンと同じ動作・フィードバック）
- **`d` キー または Delete キー** — ノートを即時削除（確認ダイアログなし）。削除後のフォーカスは下のカードに移る。下がなければ上のカードに移る。すべて削除済みならフォーカスなし状態になる

##### フォーカスなし状態からの導入

- **↑ / ↓ キー** — フィード先頭（最新）のカードにフォーカスが当たる

##### 矢印キーの有効範囲

矢印キーによるカード間移動は**カードフォーカス状態**および**フォーカスなし状態**でのみ有効。以下の状態では矢印キーは各コンテキスト固有の動作に使われる：

- 編集モード中 — CodeMirror 内のカーソル移動
- 検索バーにフォーカスがある時 — テキスト入力 / フィルタ操作

##### 編集モード中のキーボードショートカット

編集モード中 (CodeMirror 6 にフォーカス) は以下のショートカットが有効:

- **Cmd+Z / Ctrl+Z** — Undo（元に戻す）
- **Cmd+Shift+Z / Ctrl+Y** — Redo（やり直し）
- **Esc** — 自動保存して表示モードに戻る（カードフォーカスは残る）
- **Cmd+N / Ctrl+N** — 現在のノートを自動保存後、新規ノート作成

#### フィード表示

- デフォルトフィルタ：**直近7日間**のノートのみ表示
- 新しいノートが上に来る降順表示
- スクロールで過去のノートを閲覧
- **ノート間の区切りは区切り線（border/divider）ではなく空きマージンで表現する**。テーマによっては極薄の divider を補助的に許容するが、既定では罫線なしの「空白で切る」レイアウトとする（Logseq の journals フィードに倣う。書く行為に集中させ、ノート単位を"独立したドキュメント"として知覚させるため）
- **ノート数変化時のアニメーション** — ノートの作成・削除によりフィード内のノート数が変化したとき、以下のアニメーションを適用する：
  - 新規ノート：フェードインで登場
  - 削除ノート：フェードアウトで退場
  - 残存ノート：FLIP アニメーションで滑らかに位置移動
  - アニメーション中も操作をブロックしない（連続削除等を即座に受け付ける）
  - `prefers-reduced-motion: reduce` が有効な環境ではアニメーションを無効化して即時切替にフォールバックする

## 非機能要件

### テーマ対応

- ライト・ダーク両テーマに対応する
- OS のカラースキーム設定（`prefers-color-scheme`）に追従する
- テーマの手動切り替え UI は設けない

### 視認性

- カーソルおよび Markdown シンタックスハイライトが適用されるすべての構文要素（見出し・太字・斜体・リスト・コードブロック・取り消し線等）は、ライト・ダーク両テーマにおいて背景色と十分に区別できる色で表示すること
