# CoDD Drift Fix

要件テキストまたは実装テキストのスニペットを入力として、CoDD ドキュメント階層（requirements → 設計書 → 実装計画 → ソースコード）を縦断的にトレースし、ドリフト（乖離・矛盾）を検出して修正する。

## When to Use

- `codd implement` で生成されたコードが要件を無視しているとき
- 要件変更後、設計書または実装計画が追従できているか確認したいとき
- 設計書と実装計画の間に矛盾を発見したとき
- ユーザーが「この仕様が実装に反映されているか確認して」と言ったとき

## Workflow

### Step 1: 入力テキストのレイヤー特定

ユーザーが指定したテキストがどのレイヤーに属するか判定する。

| レイヤー | パス |
|---------|------|
| requirements | `docs/requirements/requirements.md` |
| 設計書 (Wave 1–4) | `docs/detailed_design/*.md`, `docs/acceptance_criteria/*.md`, `docs/adr/*.md` |
| 実装計画 (Wave 5) | `docs/plan/implementation_plan.md` |
| ソースコード | `src/`, `src-tauri/` |

判定方法: 入力テキストを grep で各レイヤーを検索し、どこに出現するかで特定する。

```bash
grep -rn "<入力テキストのキーワード>" docs/requirements/
grep -rn "<入力テキストのキーワード>" docs/detailed_design/
grep -rn "<入力テキストのキーワード>" docs/plan/
grep -rn "<入力テキストのキーワード>" src/ src-tauri/
```

### Step 2: トレース（全レイヤー検索）

入力テキストに関連するキーワードで全レイヤーを検索し、各レイヤーの記述内容を収集する。

- requirements.md: 原文を抽出
- 設計書: 関連 node_id と記述を抽出
- 実装計画: 関連 Sprint・成果物・リスク欄の記述を抽出
- ソースコード: 実際の実装を抽出（存在する場合）

### Step 3: ドリフト分析

収集した各レイヤーの記述を比較し、矛盾・欠落・意味の反転を検出する。

検出パターン:

| パターン | 例 |
|---------|-----|
| 上流に記述あり、下流で無視 | requirements に「全文表示」、実装計画に `max-height: 120px` |
| 下流で記述が追加・上書き | 設計書にない制約が実装計画に追加されている |
| 記述欠落 | 設計書には記述があるが実装計画の dependency_node_ids に含まれていない |
| 意味の反転 | 「禁止」が「必須」に変わっている |

### Step 4: 修正方向をユーザーに確認

ドリフトを検出したら、修正方向を**必ずユーザーに確認**する。

確認文のフォーマット:

```
【ドリフト検出】

■ 上流（requirements / 設計書）の記述:
  <内容>

■ 下流（実装計画 / ソースコード）の記述:
  <内容>

■ 矛盾: <矛盾の内容を1行で>

修正方向を選んでください:

A) 要件 → コード方向: 上流の記述を正として、下流（設計書/実装計画/コード）を修正する
B) コード → 要件方向: 下流の実装を正として、上流（requirements/設計書）を修正する
```

要件側を修正する B) を選ぶケースは稀であり、ユーザーが意識的に選択した場合のみ実行する。

### Step 5: 修正実行

ユーザーが方向を選択したら修正を実行する。

**A) 要件 → コード方向（下流を修正）**

1. 矛盾箇所を特定（ファイルパスと行番号）
2. 上流の記述を正として下流のドキュメントを Edit
3. ソースコードが既に存在する場合はコードも修正
4. `codd validate` で frontmatter 整合性を確認

**B) コード → 要件方向（上流を修正）**

1. ユーザーに変更内容を再確認（「要件を以下の通り変更します。よろしいですか？」）
2. requirements.md を Edit
3. `codd impact --path .` で影響波及を確認
4. 影響を受けた設計書を `codd generate --wave N --force` で再生成するかユーザーに確認

### Step 6: コミット確認

修正が完了したらコミットするか確認する。コミットする場合は CLAUDE.md のテンプレートに従う。

## HITL Gates

1. **Step 4 の方向確認**: 必ずユーザーに確認する。自動判定で修正を進めない。
2. **B) 選択時の追加確認**: 要件変更は影響が大きいため、実際の修正内容を示してから再確認する。
3. **修正後のレビュー**: 修正した全ドキュメントをユーザーが確認できるよう、変更点を明示する。

## Guardrails

- 複数のドリフトを一度に修正しない。1 件ずつ確認・修正する。
- 設計書の frontmatter（`depends_on`, `node_id` 等）は変更しない。body のみ修正する。
- `codd propagate --verify` は spec を実装に寄せる改変を混ぜるため使用しない（memory: feedback_codd_verify_drifts_to_impl.md）。
- ソースコードを修正した場合は `codd propagate --path .`（`--update` なし）で影響設計書を確認する。

## Example

```
/codd-drift-fix ノートカードは本文全文を常に表示する。カード内スクロールは設けず
```

1. grep で全レイヤーを検索
2. requirements.md: 「本文全文を常に表示する」✓
3. feed_search_design.md §4.4b: 「max-height によるプレビュー制限は行わない」✓
4. implementation_plan.md Sprint 2-3: `max-height: 120px; overflow: hidden` ← **矛盾**
5. ユーザーに A/B を確認
6. A 選択 → 実装計画を修正
